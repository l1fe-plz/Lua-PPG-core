Методы обёртки:
Т.к. обычный ModAPI часто ссылается на локальные пути (от mod.json), то пришлось поменять методы загрузки на эти:
NewTexture(string локальный_путь_от_lua.json) — загружает Texture2D.
NewSprite(string локальный_путь_от_lua.json) — загружает Sprite.
NewSound(string локальный_путь_от_lua.json) — загружает AudioClip.

А ещё при работе с MoonSharp тот плохо обрабатывает Type, и работа с компонентами превращается в ад (но для вас он упрощён, конечно же):
New(Type, аргументы...) — если в обычном C# вы бы вызвали new Vector2(1, 1), то здесь нужно писать иначе: New(Vector2, 1, 1). То же самое касается других конструкторов.

AddComponent и GetComponent... Тут ситуация сложнее. GetComponent можно вызвать, как и раньше, через строки (string): cm.GetComponent("SpriteRenderer").

Но AddComponent имеет иную структуру: AddComponent(GameObject (ОБЯЗАТЕЛЬНО GameObject!), object (далее превратится в Component)). Оба метода возвращают Component.
Из мелочей: Transform.SetParent теперь тоже работает иначе: SetParent(Transform(ребёнок), Transform(наш родитель), bool worldPositionStays). Второе значение (bool) используется для обозначения, останется ли ребёнок на своих координатах или будет сразу присвоен к нулям (true — на своём месте, false — к родителю).

Action(object функция) — преобразует функции в Action<GameObject>.

AddLuaEvents(GameObject) — вешает на ваш объект класс LuaEvents, с помощью которого можно задавать в ивенты Unity и PPG свои функции.

Немного о ядре, ядро представляет из себя простой вызов и проверки.
1. Ядро определяет, в какой папке находится, локальные моды, путь к MoonSharp.Interpreter.dll, создание списков и т.д.
2. Ядро входит в OnLoad(), тут уже ядро находит папку к стим модам и запускает консоль, создаётся экземпляр класса Lua.
3. Ядро проходит все папки с модами в поисках lua.json.
4. Начинается цикл, в котором отсеиваются все невалидные json и json парсится в LuaModMetadata.
5.1. Начинается 2 цикл, в ходе которого из путей json-файлов мы читаем все скрипты на синтаксис, при хоть одной ошибке в моде мод не запустится, также создаём для каждого мода свой экземпляр MoonSharp, в конце собираем прочитанные файлы и метадату в один список.
5.2. В создание MoonSharp мы регистрируем все нужные классы и создаём NewTexture, NewSprite, NewSound, заранее записывая в них пути (в AddLuaEvents передаём экземпляр MoonSharp), в конце отдаём MoonSharp в метадату.
6. Запускаем скрипты, которые прошли цикл ранее, все процессы после пункта 2 логируются в консоль.

Inside, we see a clear hierarchy that must be followed.:
outside of the function, it is called immediately after the core has read the scripts and run through the syntax. It is better to place global functions here.

function Main() — called when you enter one of the "chamber" (load the map), everything is created there.

function OnUnload — called at the moment the game is closed, for normal cases this function is not needed (you can not explicitly specify it in the script, like Main()), you can and should use it to save data.

Wrapper Methods:
Since the usual ModAPI often refers to local paths (from mod.json), we had to change the loading methods to these:
NewTexture(string local_path_relative_lua.json) — loads Texture2D.
NewSprite(string local_path_relative_lua.json) — loads the Sprite.
NewSound(string local_path_relative_lua.json) — loads AudioClip.

Also, when working with MoonSharp, it doesn't handle Type well, and working with components turns into hell (but it's simplified for you, of course):
New(Type, arguments...) — if in normal C# you would call new Vector2(1, 1), then here you need to write differently: New(Vector2, 1, 1). The same goes for other constructors.

AddComponent and GetComponent... The situation is more complicated here. GetComponent can be called, as before, via strings (string): cm.GetComponent("SpriteRenderer").

But AddComponent has a different structure: AddComponent(GameObject (MUST BE GameObject!), object (then it will turn into Component)). Both methods return Component.

From the little things: Transform.SetParent now also works differently: SetParent(Transform(child), Transform(our parent), bool worldPositionStays). The second value (bool) is used to indicate whether the child will remain at its coordinates or will be immediately assigned to zeros (true — in its place, false — to the parent).

Action(object function) — converts functions into Action<GameObject>.

AddLuaEvents(GameObject) — attaches the LuaEvents class to your object, with which you can set your own functions in Unity and PPG events.

LuaEvents:
AddUpdate(object function)
AddFixedUpdate(object function)
AddLateUpdate(object function)
AddStart(object function)
AddAwake(object function)
AddCollisionEnter(object function with value) — function(coll) (returns the collision of the object that the GM encountered).
AddCollisionExit(object function_with_value)
AddUse(object function_with_value)
AddShot(object function_with_value)
AddDamage(object function_with_value)
AddBreak(object function_with_value)
AddStab(object function_with_value)

A little bit about the core, the core is a simple call and verification.
1. The core determines in which folder it is located, the local mods, the path to MoonSharp.Interpreter.dll , creating lists, etc.
2. The core enters OnLoad(), then the core finds the folder for steam mods and launches the console, an instance of the Lua class is created.
3. The core goes through all the folders with mods in search of lua.json.
4. A cycle begins in which all invalid json is filtered out and json is parsed to LuaModMetadata.
5.1. The 2nd cycle begins, during which we read all the syntax scripts from the paths of the json files, if there is at least one error in the mod, the mod will not start, we also create our own MoonSharp instance for each mod, at the end we collect the read files and metadata into one list.
5.2. To create MoonSharp, we register all the necessary classes and create NewTexture, NewSprite, NewSound, writing paths in them in advance (we pass an instance of MoonSharp to AddLuaEvents), at the end we give MoonSharp to the metadata.
6. Run the scripts that went through the cycle earlier, all processes after step 2 are logged into the console.
